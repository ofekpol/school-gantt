    **Step 1 — Create the directory:** `mkdir -p .github/workflows`

    **Step 2 — Create `.github/workflows/ci.yml` verbatim:**

    ```yaml
    name: CI

    on:
      pull_request:
        branches: ["**"]
      push:
        branches: ["main"]

    concurrency:
      group: ci-${{ github.ref }}
      cancel-in-progress: true

    permissions:
      contents: read

    jobs:
      ci:
        name: Lint, Typecheck, Test, E2E
        runs-on: ubuntu-latest
        timeout-minutes: 20

        steps:
          - name: Checkout
            uses: actions/checkout@v4

          - name: Setup pnpm
            uses: pnpm/action-setup@v4
            with:
              version: 9.12.0

          - name: Setup Node
            uses: actions/setup-node@v4
            with:
              node-version-file: ".nvmrc"
              cache: "pnpm"

          - name: Install dependencies
            run: pnpm install --frozen-lockfile

          - name: Lint (ESLint)
            run: pnpm lint

          - name: Typecheck (tsc --noEmit)
            run: pnpm tsc --noEmit

          - name: Unit tests (Vitest)
            run: pnpm test

          - name: Build
            run: pnpm build

          - name: Install Playwright browsers
            run: pnpm dlx playwright install chromium --with-deps

          - name: E2E tests (Playwright)
            run: pnpm playwright
            env:
              CI: "true"

          - name: Upload Playwright report on failure
            if: failure()
            uses: actions/upload-artifact@v4
            with:
              name: playwright-report
              path: playwright-report/
              retention-days: 7
    ```

    **Why this order (D-05):** lint catches style violations cheapest, typecheck catches type errors before runtime, unit tests catch logic regressions, build proves the app compiles, e2e catches integration bugs. Each step is a gate — first failure aborts the job.

    **Why we add a build step** (not in D-05 explicitly): the Playwright `webServer` runs `pnpm dev` which is fine, but a production build catches issues that dev mode hides (RSC serialization, treeshaking, edge runtime). It costs ~30s and prevents a class of "works in dev, breaks in prod" bugs. This is Claude&apos;s discretion within D-05&apos;s spirit.

    **Why frozen-lockfile:** ensures CI installs exactly what was committed in `pnpm-lock.yaml` — no surprise version drift.

    **Why concurrency cancel:** if a PR gets two pushes in quick succession, only the latest runs to completion (saves CI minutes).

    Do NOT add `workflow_dispatch` — keep triggers strictly per D-03.
    Do NOT add a `push: branches: ['**']` trigger — that would run CI on feature pushes without a PR, violating D-03.
  </action>
